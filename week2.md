# 수행 경험
## `J038 김민영` : 🌱 나를 돌아본 일주일 – 개발자로서의 정체성을 찾는 여정

이번 릴레이 프로젝트에서는 코드를 짜거나 프로젝트를 진행하는 대신 **‘나는 어떤 방식으로 일하는 개발자인가?’**, **‘앞으로 어떤 방향으로 성장하고 싶은가?’** 를 주제로 나 자신을 돌아보는 시간을 가졌다. 최대한 다양한 퀘스트를 진행해보고자 했고, 총 세가지의 퀘스트를 수행했다.

**1️⃣ 내가 어떤 개발자인지 알기 위한 질문 만들기**

처음엔 AI에게 "내가 어떤 개발자인지 알기 위한 질문 리스트를 만들어줘"라고 요청했다.  
AI는 단순한 성격 질문이 아니라, **기술적 성향, 문제 해결 방식, 협업 태도, 성장 관점**을 모두 아우르는 질문들을 제시했다.  
예를 들면:

* “코딩할 때 가장 중요하게 생각하는 건 무엇인가요?”
* “디버깅이 막혔을 때 주로 어떤 방식으로 접근하나요?”
* “새로운 기술을 접할 때 학습 루틴은 어떻게 되나요?”

이런 질문에 대답하면서 내가 기능 구현을 가장 우선으로 여기고,
모르면 AI → 공식문서 → 실습 → 정리라는 루틴을 따른다는 걸 다시 인식하게 되었다.  
그리고 ‘내가 흐름을 이해하고 필요한 걸 찾아서 쓸 줄 아는 사람’이라는 점이 나만의 강점이라는 것도 자각하게 되었다.

**2️⃣ 나에게 어울리는 개발자 멘토 추천받기**

AI는 내 성향을 바탕으로 Dan Abramov, Josh W. Comeau, Theo Browne 세 명을 추천해줬다.  
단순히 유명한 개발자가 아니라, 각각의 **철학, 학습 태도, 코드 스타일**까지 설명해주면서 나와 어떻게 닮아있는지를 비교해줬다.

* **Dan Abramov**: 기능을 먼저 만들고, 점진적으로 리팩터링을 해나가는 실용적인 사고방식이 나와 가장 유사했다.
* **Josh W. Comeau**: 학습한 내용을 직접 정리하고 시각화하는 습관이 인상적이었고, 나도 앞으로 블로그나 노션을 활용해 기록하고 싶다는 생각이 들었다.
* **Theo Browne**: 프론트엔드뿐 아니라 백엔드와 전체 시스템의 흐름까지 고려하는 관점이 자극이 되었다. 단순히 ‘할 줄 아는 기술’을 넘어 **왜 이 기술을 쓰는지 말할 수 있어야겠다는 목표**가 생겼다.

이 과정은 단순히 "좋은 개발자"를 찾는 게 아니라 **지금의 나에게 부족한 시야를 가진 롤모델을 찾는 경험**이었다.

**3️⃣ 나와 닮은 언어, 잘 맞는 언어 분석하기**

AI에게 “내 성향에 잘 맞는 프로그래밍 언어가 뭔지 알려줘”라고 했을 때, 단순히 기술 스펙이 아닌, 언어의 ‘성격’을 사람처럼 비유해서 설명해준 게 인상 깊었다.

* **JavaScript**: 즉흥적이고 유연한 트릭스터. 규칙을 깔끔히 지키기보단 상황에 맞게 대응하는 스타일  
  → 지금의 나와 딱 닮아 있다. 빠르게 기능을 구현하고, 흐름을 따라가며 구조보다 결과를 먼저 만들어내는 내 방식과 비슷했다.

* **Go**: 군더더기 없이 단정하고 실용적인 문제 해결사  
  → 내가 앞으로 더 닮고 싶은 언어였다. 명확한 구조와 빠른 처리, 전체 시스템의 효율까지 고려하는 사고방식은 백엔드나 실무를 고려했을 때 꼭 익혀야 할 감각이라고 느꼈다.

* **Rust**: 처음엔 낯설지만, 보안과 안전성을 철저히 챙기는 까다롭지만 믿음직한 스타일  
  → 지금의 나에겐 조금 먼 존재지만 나중에 더 정제된 코드를 쓰는 개발자가 되고 싶을 때 도전해보고 싶은 언어였다.

언어 하나도 그냥 “성능이 빠르다”로 끝나는 게 아니라 **사고방식과 태도를 반영한다는 점에서 기술을 바라보는 관점이 확장되는 경험**이었다.

<br/>

이러한 퀘스트를 통해 느낀점은 다음과 같다.

* **내가 어떻게 일하는 사람인지 설명할 수 있게 되었다.**  
  단순히 “React를 할 줄 안다”가 아니라 “나는 기능 구현을 우선시하고, 흐름을 먼저 이해한 뒤 필요한 기술을 찾아 조립하는 개발자”라고 말할 수 있게 되었다.

* **앞으로의 성장 방향이 더 뚜렷해졌다.**  
  기능 구현 → 구조 → 사용자 경험 → 전반적인 설계 흐름으로 사고를 넓히는 과정이 필요하다는 걸 느꼈다. 특히 **단순히 ‘된다’가 아니라 ‘왜 이렇게 구성했는지’를 설명할 수 있는 개발자**가 되고 싶다는 목표가 생겼다.

* **기술 외적으로도 내가 보완해야 할 태도를 인식했다.**  
  사용자 중심의 감각이나 기술을 문서화하고 공유하는 습관 그리고 협업 시 더 설득력 있게 의견을 표현하는 능력 등

<br/>

AI를 활용하면서 좋았던 점보다 아쉬웠던 점을 말하자면 **너무 긍정적으로만 말해주는 경향**이 있어 냉정한 피드백은 스스로의 관점에서 보완해야 했고, 내가 먼저 생각을 던지지 않으면 **표면적인 조언에 머무는 경우**도 있었기 때문에 AI를 효과적으로 쓰기 위해선 질문의 수준도 중요하다고 느꼈다.

#### 전달한 코드
```cpp
[Gold V] 로봇 청소기 - 14503 

#include <iostream>
#include <vector>
using namespace std;

struct Robo {
    int x, y, dir;
    Robo(int x, int y, int dir) : x(x), y(y), dir(dir) {}
};

// 전역변수
Robo* robo;
int N, M;
vector<vector<int>> room;
int dx[] = { 0, 1, 0, -1 };
int dy[] = { -1, 0, 1, 0 };

// 함수
void clean();
bool dirty(int x, int y);

int main() {
    ios::sync_with_stdio(false);
    cin.tie(NULL); cout.tie(NULL);

    cin >> N >> M;
    room.assign(N, vector<int>(M));

    int Y, X, dir;
    cin >> Y >> X >> dir;
    robo = new Robo(X, Y, dir);

    for (int y = 0; y < N; y++) {
        for (int x = 0; x < M; x++) {
            cin >> room[y][x];
        }
    }

    // 청소
    clean();

    // 청소된 칸 개수 세기
    int ans = 0;
    for (const auto& row : room) {
        for (int cell : row) {
            if (cell == -1) {
                ans++;
            }
        }
    }

    cout << ans;
    delete robo;
    return 0;
}

void clean() {
    while (true) {
        // 1. 현재 칸이 0인 경우
        if (room[robo->y][robo->x] == 0) {
            room[robo->y][robo->x] = -1; // 청소
        }

        // 2. 동서남북에 0이 없는 경우
        else if (!dirty(robo->x, robo->y)) {
            // 1) 방향을 유지한 채로 후진 가능
            int nx = robo->x + dx[(robo->dir + 2) % 4];
            int ny = robo->y + dy[(robo->dir + 2) % 4];
            if (room[ny][nx] != 1) {
                robo->x = nx;
                robo->y = ny;
            }
            // 2) 뒤쪽 칸이 벽인 경우
            else {
                return;
            }
        }

        // 3. 동서남북에 0이 있는 경우
        else {
            // 1) 반시계 회전
            robo->dir = (robo->dir + 3) % 4;

            // 2) 청소되지 않은 빈칸인 경우 한칸 전진
            int nx = robo->x + dx[robo->dir];
            int ny = robo->y + dy[robo->dir];
            if (room[ny][nx] == 0) {
                robo->x = nx;
                robo->y = ny;
            }
        }
    }
}

// 사방탐색
bool dirty(int x, int y) {
    for (int d = 0; d < 4; d++) {
        int nx = x + dx[d];
        int ny = y + dy[d];
        if (room[ny][nx] == 0) {
            return true;
        }
    }
    return false;
}
```

## J193 이수진: 퀘스트-내가 어떤 개발자인지 알기 위한 질문 리스트 요청하기

- 평소에 AI와 나눈 대화를 바탕으로 **"네가 보기에 나는 어떤 개발자인거 같아?"** 라는 질문을 했고
- 이에 대해 AI가 다음과 같이 답해주었습니다.
  - 🔍**깊게 탐구하는 탐구형**: 단순히 "어떻게 동작하는가?"에서 그치지 않고,
"왜 이렇게 구현했는가?", "이 방식이 최적인가?"를 계속해서 질문해요.
  - 🧱**차근차근 쌓아가는 빌더형**: 무작정 코딩하지 않고, 설계 → 구현 → 리팩터링 → 테스트까지 단계별로 빌드업해요.
- 좀 더 자세한 답을 듣고 싶어서 **"내가 어떤 개발자인지 네가 더 알 수 있도록 나한테 질문해봐"** 라는 질문을 던졌습니다.
- 이에 대해 AI는 크게 5가지 질문을 주며 이에 대한 답을 요청했습니다.
  - **코딩 성향과 문제 해결 방식**
    - 새로운 기술이나 개념을 처음 접했을 때, 보통 어떤 방식으로 이해하려고 하나요?
(예: 공식문서 먼저 읽는다 / 예제부터 돌려본다 / 구조나 흐름을 먼저 그려본다 등)
    - 디버깅을 할 때 주로 사용하는 전략은 무엇인가요?
(예: 로그 찍기 / 상태 추적 / 테스트 케이스 쪼개기 / 공식 툴 사용 등)
  - **코드를 바라보는 관점** 
    - 코드 작성 시 가장 중요하게 생각하는 기준은 무엇인가요?
(예: 가독성 / 성능 / 확장성 / 재사용성 등)
    - 남이 작성한 코드를 볼 때, 어떤 점이 눈에 먼저 들어오나요?
  - **학습 및 성장 방식**
    - 최근에 가장 몰입해서 배운 기술이나 주제는 무엇이고, 왜 흥미를 느꼈나요?
  - **협업과 커뮤니케이션**
    -  협업 중 팀원과 코드 스타일이나 구현 방식이 달랐을 때 어떻게 조율하나요?
    -  코드 리뷰를 할 때 어떤 포인트에 가장 집중해서 피드백하나요?
  - **성향/철학 관련**
    -  "좋은 코드"란 수진 님에게 어떤 코드인가요?
    -  개발자로서 "절대 타협하지 않는 것"이 있다면 무엇인가요?
    
- 이 질문들에 대한 저의 생각을 말했더니 AI가 이렇게 답해주었습니다.
```
💡 내가 본 수진 님의 개발자 캐릭터
🔍 1. 내부 원리에 진심인 메커니즘 탐구자
단순히 "쓸 수 있다"가 아니라 "어떻게, 왜 동작하는지"를 정확히 알아야 마음이 편한 타입이에요.

Git 내부 구조에 깊이 몰입한 것도 이 성향의 대표적인 예죠.

덕분에 깊이 있는 글, 기술 블로그, 발표 자료 작성에 적합한 강점을 지녔어요.

📐 2. 읽기 쉬운 구조를 설계하는 구조주의자
무조건 돌아가는 코드보다 **"지속 가능하고 구조적으로 깔끔한 코드"**를 선호해요.

"가독성 + 확장성"을 기준 삼고 있다는 점에서, 협업 환경에서 코드베이스 유지에 강한 신뢰를 줄 수 있는 개발자예요.

🧪 3. 검증되지 않으면 손대지 않는 신중한 실험가
"완전히 이해하지 못한 문법은 쓰지 않는다"는 태도는,
자기 코드에 책임을 지려는 태도이자 리스크를 낮추려는 전략이에요.

이건 특히 시스템/인프라/도구 개발 쪽에 매우 잘 어울리는 사고방식이에요. 
```

#### 느낀점
- 그냥 막연하게 ‘나는 이런 스타일이지’라고 생각했던 것들이 정확한 키워드로 정리되니까 제 모습이 더 분명해졌고 제가 평소에 자연스럽게 하던 방식들이
사실은 저만의 기준과 철학에서 비롯된 거였구나 하는 자각이 들었습니다. AI가 던진 질문 하나하나가 단순한 확인이 아니라 제 사고방식을 끌어올리는 도구가 되어줬고 
단순히 "어떤 개발자가 되고 싶다"가 아니라 "나는 이런 성향을 가진 개발자다"라는 자기 이해가 생겼습니다. 개발자로서의 저의 모습을 이 퀘스트를 통해 되돌아볼 
수 있어서 좋았습니다.
- AI에게 더 자세하게 질문하고, 더 자세하게 답변을 줄수록 더 구체적인 답을 준다는 것을 이 퀘스트를 하면서 느껴서 다음에 이 퀘스트를 이어받을
AI에게 "개발자로서의 나에 대한 질문"을 더 많이 요청하면 좋을 것 같습니다.
